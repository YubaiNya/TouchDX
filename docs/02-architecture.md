# 02. 系统整体架构 (Architecture)

TouchDX 不是一个单一的软件，而是一套分布式的系统。为了实现从物理触摸到游戏内判定的毫秒级同步，我们将其拆分为三个核心子项目，外加一条高速通信总线。

## 架构概览

```text
[ 安卓设备 (平板/手机) ]                    [ PC (Windows) ]
       |                                          |
       |                                          v
+------------------+                    +---------------------+
|   TouchDX App    |                    |  maimai DX (游戏)   |
|  (MaiNative)     |                    |                     |
|                  |     网络通信        |  +---------------+  |
| - SVG 解析与渲染  | -----------------> |  | TouchDX-Mod   |  |
| - 原生触摸事件处理|   (ADB 有线转发)     |  | (MelonLoader) |  |
| - 按键状态位掩码  |  TCP Port: 4321    |  +---------------+  |
+------------------+                    +---------------------+
                                                  ^
                                                  | 
                                                  v
                                        +---------------------+
                                        | PCDiagnosticTool    |
                                        | (独立诊断抓包工具)    |
                                        +---------------------+
```

## 核心组件说明

### 1. MaiNative (安卓客户端)
这是玩家直接交互的终端。它使用 Android 原生的 SDK 开发，保证了最低的触控响应延迟和最高的硬件兼容性。
*   **职责**：呈现按键区域、捕捉多点触控、将坐标映射为 maimai 的标准按键状态（A-E区、外圈键），并将这些状态压缩为一个 64 位的 `ulong` 掩码发送给 PC。

### 2. TouchDX-Mod (PC 注入端)
这是系统的“大脑中枢”。它是一个运行在 MelonLoader 环境下的动态链接库 (`.dll`)。
*   **职责**：在本地 `4321` 端口开启 TCP 服务器。接收到掩码后，通过 `user32.dll` 模拟实体键盘的按下/抬起（如 `W, E, D, C`），同时使用 `Harmony` 框架劫持游戏内的 `InputManager` 和 `MeshButton`，将触摸状态直接写入游戏的运行内存中。

### 3. PCDiagnosticTool (诊断工具)
由于网络环境和 Mod 加载存在诸多不确定性，诊断工具起到了“听诊器”的作用。
*   **职责**：同样在 `4321` 端口开启服务器（不能与游戏同时运行），捕获并解析 Android 端发送的掩码，可视化展示当前的输入状态，用于排查网络断连或触控失灵问题。

## 为什么是 ADB 有线转发？
普通的 Wi-Fi 局域网传输在面临音游 16.6ms (60fps) 或更短的判定窗口时，往往会出现严重的 Jitter (抖动) 现象。通过 `adb reverse tcp:4321 tcp:4321`，我们利用 USB 数据线建立了一条物理级的高速通道，这使得网络传输延迟几乎可以忽略不计。
