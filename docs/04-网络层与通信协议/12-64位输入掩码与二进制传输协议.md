# 12. 64 位输入掩码与二进制传输协议

在 120Hz 甚至更高的报点率下，如果使用 JSON 或字符串来传输按键状态（比如 `{"A1": true, "B2": false}`），网络开销和序列化/反序列化的 CPU 损耗将是灾难性的。

为了追求极限性能，TouchDX 采用了最为硬核的位运算压缩协议。

## 数据包结构：极致的 8 字节

整个触摸面板和物理按键的状态，被完美地塞进了一个 `long`（在 C# 中是 `ulong`，64 位无符号整型）中。这刚好是 **8 个字节**。

在 Android 端 (Java)：
```java
// 声明一个 long 类型的变量来保存所有状态
private long currentState = 0L;

// 当某个按键被按下时，利用位运算将特定的 bit 位置为 1
currentState = currentState | (1L << buttonIndex);

// 当某个按键抬起时，将对应的 bit 位置为 0
currentState = currentState & ~(1L << buttonIndex);
```

## 按键索引分配表 (Bit Index)

这 64 个位置被精心分配给了 maimai DX 的所有交互元素：

| Bit Index (位索引) | 对应区域/按键 | 用途说明 |
| :--- | :--- | :--- |
| `0 - 7` | `Btn1` - `Btn8` | 1P 的 8 个外围物理按键 |
| `8 - 15` | `A1` - `A8` | 触摸屏最外圈 |
| `16 - 23` | `B1` - `B8` | 触摸屏中间圈 |
| `24 - 25` | `C1`, `C2` | 触摸屏最内圈 (中心) |
| `26 - 33` | `D1` - `D8` | 触摸屏星滑外圈 |
| `34 - 41` | `E1` - `E8` | 触摸屏星滑内圈 |
| `42` | `Select` | 选择/确认 控制键 |
| `43` | `Test` | 测试模式 控制键 |
| `44` | `Service` | 服务菜单 控制键 |
| `45` | `Coin` | 投币键 |

## 网络层的 Little-Endian 转换

由于网络字节序的标准，以及 Java 与 C# 底层的差异。在发送数据前，我们需要确保字节序的统一。我们使用了 `ByteBuffer` 并显式指定了小端序：

```java
ByteBuffer objectBuffer = ByteBuffer.allocate(8);
objectBuffer.order(ByteOrder.LITTLE_ENDIAN);
objectBuffer.putLong(currentState);
socketChannel.write(objectBuffer);
```

在 PC 端 (C#)，直接将收到的 8 字节数组转回 ulong：
```csharp
ulong newState = BitConverter.ToUInt64(buffer, 0);
```

这种设计使得数据包的体积达到了理论最小值，且两端的编解码只需一条 CPU 指令，延迟被压榨到了极限。
