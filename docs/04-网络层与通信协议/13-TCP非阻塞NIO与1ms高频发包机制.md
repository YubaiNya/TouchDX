# 13. TCP 非阻塞 NIO 与 1ms 高频发包机制

在音游这种对时序要求极度苛刻的场景下，传统的阻塞式 (Blocking) 网络 I/O 会带来难以预料的卡顿。TouchDX 的安卓网络层使用了 Java NIO (New I/O) 包中的 `SocketChannel` 进行重构。

## 非阻塞写入 (Non-blocking Write)

在 `MaimaiTouchClient.java` 的主循环中：

```java
this.socketChannel.configureBlocking(false); // 切换为非阻塞模式

// ... 在循环中 ...
try {
    this.socketChannel.write(objectBuffer);
} catch (Exception e) {
    // 处理异常
}
```

*   **阻塞模式的缺点**：如果底层的 TCP 发送缓冲区满了（比如网络突然拥堵了一瞬间），调用 `write()` 的线程就会被挂起休眠，直到缓冲区有空间。这会导致整个发送循环停滞，使得后续最新的按键状态无法及时处理。
*   **非阻塞模式的优势**：在非阻塞模式下，如果缓冲区满了，`write()` 会立即返回 `0`（表示写入了 0 字节），而不会阻塞线程。虽然这意味着这一个状态包被“丢弃”了，但对于高频采样的输入来说，这是**绝对正确**的选择！因为几毫秒后循环就会跑完下一圈，生成包含最新按键状态的新数据包。宁可丢弃旧的过期数据，也绝不让线程卡住。

## 极限睡眠频率：16.6ms 还是更低？

普通的 60 帧游戏，一帧的计算时间约为 16.6 毫秒。为了确保 PC 端在每一帧都能拿到尽可能新鲜的按键状态，我们的发包频率必须远高于游戏的刷新率。

```java
Thread.sleep(4L, 166666); // 休眠 4.16 毫秒
```

这里代码利用了纳秒级别的精细控制，让网络发送线程每 4.16 毫秒左右运行一次。这相当于以接近 **240Hz** 的频率疯狂“刷屏”发送当前按键的快照。配合 TCP 的 `NoDelay = true`（关闭 Nagle 算法避免包合并延迟），让 Android 到 PC 的通信链路始终处于亢奋的巅峰状态。
